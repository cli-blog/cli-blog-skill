# REST API

Use direct HTTP for browser delivery, non-Node backends, or callers that explicitly need the wire contract.

## Contents

- Contract and key capabilities
- Public endpoints
- Published-content reads
- Drafts and media uploads
- Pagination and filters
- Error handling

## Contract

- Base URL: `https://api.cli-blog.com`
- Authentication header: `x-api-key`
- API version prefix: `/v1`
- Reads: `GET`
- Creates and updates: `POST`
- Deletes: `DELETE`
- Non-file writes: `application/json`
- Media upload: `multipart/form-data`

Put filters, sorting, pagination, locale, fields, and includes in query parameters. Put stable identifiers or locale-scoped slugs in paths. Never put API keys in URLs or bodies.

## Key Capabilities

Use a public key for published posts, public authors, localized categories and tags, feed XML, and sitemap XML. Public post responses may expand referenced media with `include=media`.

Use a permission-scoped private key for editorial states, writes, direct media inventory, revisions, and redirects. Publication settings, billing, members, organizations, and API-key administration remain signed-in dashboard operations.

| Resource | Public key | Private key with permission |
| --- | --- | --- |
| Posts | Read published delivery | Read editorial states; create, update, publish, schedule, archive, delete |
| Authors | Read | Create, read, update, delete |
| Categories and tags | Read | Create, read, update, delete |
| Media | Read only through published post includes | Upload, list, retrieve, update metadata, delete |
| Revisions and redirects | No direct access | Read with post permission |
| Feed and sitemap | Read | Read |
| Locales | No key required at the HTTP route | Read |

## Public Endpoints

```text
GET, POST       /v1/posts
GET, POST, DELETE /v1/posts/{id}
GET             /v1/posts/{id}/revisions
GET             /v1/posts/{id}/revisions/{revisionId}
GET             /v1/posts/slug-redirects/{slug}

GET, POST       /v1/authors
GET, POST, DELETE /v1/authors/{id}

GET, POST       /v1/media
GET, POST, DELETE /v1/media/{id}

GET, POST       /v1/categories
GET, POST, DELETE /v1/categories/{id}
GET, POST       /v1/tags
GET, POST, DELETE /v1/tags/{id}

GET             /v1/locales
GET             /v1/feed
GET             /v1/sitemap
```

`{id}` accepts a slug where the resource documents slug lookup. Pass `locale` when a slug can identify different localized rows.

## Read Published Content

Request the smallest useful delivery shape:

```sh
curl --get "https://api.cli-blog.com/v1/posts" \
  --header "x-api-key: $CLI_BLOG_PUBLIC_API_KEY" \
  --data-urlencode "status=published" \
  --data-urlencode "locale=en-US" \
  --data-urlencode "fields=summary,seo" \
  --data-urlencode "include=authors,categories,tags" \
  --data-urlencode "limit=20"
```

Retrieve a detail page by locale-scoped slug:

```sh
curl --get "https://api.cli-blog.com/v1/posts/how-we-ship-release-notes" \
  --header "x-api-key: $CLI_BLOG_PUBLIC_API_KEY" \
  --data-urlencode "locale=en-US" \
  --data-urlencode "fields=summary,content,seo" \
  --data-urlencode "include=authors,categories,tags,media,translations"
```

## Create A Draft

Use JSON and stable related-resource IDs:

```sh
curl "https://api.cli-blog.com/v1/posts" \
  --header "x-api-key: $CLI_BLOG_PRIVATE_API_KEY" \
  --header "content-type: application/json" \
  --data '{
    "title": "How we ship release notes",
    "locale": "en-US",
    "status": "draft",
    "excerpt": "A repeatable draft, review, and publish workflow.",
    "body_markdown": "## Draft\n\nStart with the reader outcome.",
    "author_profile_ids": ["author_123"],
    "category_ids": ["category_123"],
    "tag_ids": ["tag_123"],
    "featured_media_asset_id": "media_123",
    "media_asset_ids": ["media_123"],
    "seo_description": "A practical release-note publishing workflow."
  }'
```

Publish through the canonical update route after approval:

```sh
curl "https://api.cli-blog.com/v1/posts/post_123" \
  --header "x-api-key: $CLI_BLOG_PRIVATE_API_KEY" \
  --header "content-type: application/json" \
  --data '{"status":"published","expected_version":3}'
```

## Upload Media

Send the file and optional editorial metadata as multipart fields. Do not set `content-type` manually; let the HTTP client add the multipart boundary.

```sh
curl "https://api.cli-blog.com/v1/media" \
  --header "x-api-key: $CLI_BLOG_PRIVATE_API_KEY" \
  --form "file=@./release-workflow.png" \
  --form "alt_text=Release workflow from draft through publishing" \
  --form "caption=The editorial release workflow"
```

The response URL and storage metadata are generated by Cli Blog. Update only `alt_text`, `caption`, or `metadata` through `POST /v1/media/{id}`.

## Paginate And Filter

Follow `next_cursor` with the same query for stable traversal. Use CSV values or repeated parameters according to the caller's URL library; the API accepts documented arrays and comma-separated controls.

```sh
curl --get "https://api.cli-blog.com/v1/posts" \
  --header "x-api-key: $CLI_BLOG_PUBLIC_API_KEY" \
  --data-urlencode "status=published" \
  --data-urlencode "category_slug=engineering,releases" \
  --data-urlencode "category_match=all" \
  --data-urlencode "exclude_tag_slug=internal" \
  --data-urlencode "sort=published_at" \
  --data-urlencode "direction=desc" \
  --data-urlencode "limit=100"
```

Use `page` and optional `per_page` only for exact totals. Do not send `limit` or `after` in the same request.

## Handle Errors

Expect a structured body with an error code, message, and optional parameter:

```json
{
  "error": {
    "code": "version_conflict",
    "message": "Post version conflict",
    "param": "expected_version"
  }
}
```

Capture the HTTP status and `x-request-id` response header. On `429`, honor `Retry-After` when present. Do not retry validation, permission, version-conflict, or plan-limit errors without changing the request or workflow.

Retry safe reads with bounded backoff for transient network, timeout, `425`, and `5xx` failures. Do not retry content writes automatically unless the surrounding workflow can prove whether the first attempt committed; the public content API does not expose an idempotency-key contract.

Consult `https://cli-blog.com/docs/reference/endpoints` when an agent needs the exhaustive schema for one operation.
